\section{Implementation and Deployment}

\subsection{Technology Stack}

The system is implemented as a modular web application. Each component is responsible for a specific part of the legal question-answering pipeline, from corpus preparation to user interaction.

The backend is built with FastAPI~\cite{fastapidocs}. It exposes the main chat endpoint, handles authenticated requests, connects to the retrieval modules, and returns answers with citations. Qdrant is used as the vector database for hybrid retrieval, while Neo4j stores the legal graph used for hierarchy and reference-based expansion. The frontend is developed with React, Vite, and TypeScript~\cite{reactdocs,vitedocs}. Supabase provides user authentication, user profiles, conversation storage, message storage, and trace data~\cite{supabasedocs}.

For answer generation, the deployed system uses Groq as the configured language-model provider~\cite{groqdocs}. In the final reproducible evaluation, deterministic extractive generation is used instead of provider-based generation, so that the benchmark results do not depend on external model availability or sampling behavior.

\subsection{Offline Corpus, Index, and Graph Build}

The offline build prepares the legal knowledge base before the system is used by end users. This stage starts from curated Vietnamese legal documents and converts them into structured evidence chunks.

Each chunk contains the legal text, citation information, document metadata, legal coordinates, and other retrieval-related fields. The system then builds two complementary representations from these chunks. The first representation is the Qdrant hybrid index, which stores dense vectors and sparse retrieval signals. The second representation is the Neo4j legal graph, which stores relationships between legal documents, articles, clauses, points, appendices, topics, and related provisions.

This offline build is important because it keeps expensive processing outside the online request path. It also ensures that the runtime system can retrieve legal evidence quickly while still preserving legal structure.

\subsection{Backend Runtime and Chat Flow}

The FastAPI backend is the main integration layer of the system. It receives user messages from the frontend, authenticates requests, retrieves legal evidence, calls the answer generation module, validates citations, and stores the conversation result.

The main chat flow starts when a user sends a question. The backend first checks the user session through Supabase authentication. It then sends the question to the retrieval pipeline. The retriever searches the Qdrant index, expands evidence through Neo4j when needed, and returns a set of selected legal contexts.

After retrieval, the backend passes the selected contexts to the answer generation module. If the system cannot find enough usable evidence, it returns an insufficient-context response instead of generating an unsupported legal answer. This behavior keeps answer generation dependent on retrieved legal sources.

The backend also stores useful trace information, such as the retrieved contexts, selected citations, latency, and insufficient-context status. These traces help analyze system behavior during evaluation and debugging.

\subsection{Answer Generation and Safety}

The answer generation module is responsible for producing a response from the selected legal contexts. In the deployed application, Groq is used as the configured language-model provider for natural-language answers. The prompt instructs the model to answer only from the retrieved legal evidence and to include cited legal bases.

However, answer generation alone is not considered sufficient for a legal information system. After generation, the citation validator checks whether the cited legal bases actually appear in the retrieved evidence. It also detects unsupported article numbers and insufficient-context cases, which helps reduce hallucinated or unsupported legal references.

The system also includes scope control. Some questions require legal documents that are not included in the indexed corpus, such as regional minimum wage, insurance contribution rates, administrative penalties, foreign-worker work permits, trade-union fees, or personal income tax. When the system detects that the available corpus is insufficient, it should refuse or mark the answer as insufficient-context.

This design follows a conservative safety principle. It is better for the assistant to return a limited answer or refuse a question than to provide a fluent answer with unsupported legal citations.

\subsection{Frontend and Deployment Architecture}

The frontend is implemented with React, Vite, and TypeScript. It provides the user interface for asking questions, viewing answers, checking citations, and reviewing conversation history. The frontend communicates with the FastAPI backend through HTTP requests.

Authentication is handled through Supabase Auth, including Google OAuth. After login, the frontend sends the Supabase access token to the backend in the request header. This allows the backend to protect chat and conversation routes.

In deployment, the frontend can be hosted on Vercel, while the FastAPI backend can be hosted separately. Supabase stores user profiles, conversations, messages, and chat traces. Qdrant stores the retrieval index, and Neo4j stores the legal graph. This separation makes the system easier to maintain because each service has a clear role.

\subsection{Evaluation Reproducibility}

The implementation also includes a reproducible evaluation mode. This mode uses fixed benchmark files, a fixed index manifest, and structured output files. The purpose is to make the reported results repeatable.

The final evaluation does not depend on random language-model sampling. Instead, it uses deterministic extractive generation to evaluate whether the retrieval and citation-grounding pipeline works as expected. This choice makes the evaluation less fluent than a full LLM-based answer in some cases, but it provides more stable benchmark results.

This separation between deployed runtime generation and deterministic evaluation is important for RAG systems whose results may otherwise vary with retrieval settings, prompts, or model sampling. The deployed system can use a configured language-model provider for user-facing answers, while the benchmark can remain reproducible for thesis evaluation.
