\chapter{Graph Schema}
\label{app:graph-schema}

This appendix documents the labels, relationship types, properties, and constraints implemented within the Neo4j relational knowledge graph.

\section{Ontology Labels and Relationships}
\begin{itemize}
  \item \textbf{Node Labels}: \texttt{Legal\_Document}, \texttt{Legal\_Article}, \texttt{Legal\_Clause}, \texttt{Legal\_Point}, \texttt{Legal\_Appendix}, \texttt{Evidence\_Chunk}, \texttt{Legal\_Topic}, \texttt{Legal\_Actor}, and \texttt{Legal\_IssueType}.
  \item \textbf{Relationship Types}: 
    \begin{itemize}
      \item \textit{Structural Hierarchy}: \texttt{HAS\_ARTICLE}, \texttt{HAS\_CLAUSE}, \texttt{HAS\_POINT}, \texttt{HAS\_APPENDIX}.
      \item \textit{Source Link}: \texttt{HAS\_SOURCE\_CHUNK} (connecting leaf-level legal anchors to \texttt{Evidence\_Chunk} nodes).
      \item \textit{Cross-Reference}: \texttt{REFERENCES} (citations), \texttt{DETAILS} (qualifications), and \texttt{GUIDED\_BY} (administrative delegations).
      \item \textit{Taxonomic Metadata}: \texttt{MENTIONS\_TOPIC}, \texttt{APPLIES\_TO\_ACTOR}, \texttt{HAS\_ISSUE\_TYPE}.
      \item \textit{Normative Hierarchy}: \texttt{SUPERIOR\_TO}, \texttt{MUST\_COMPLY\_WITH}.
    \end{itemize}
\end{itemize}

\section{Constraints and Indexes}
To ensure structural integrity and quick lookups, unique node constraints are applied to properties:
\begin{ThesisCode}
CREATE CONSTRAINT legal_node_node_id IF NOT EXISTS
FOR (n:LegalNode) REQUIRE n.node_id IS UNIQUE;

CREATE CONSTRAINT evidence_chunk_chunk_id_unique IF NOT EXISTS
FOR (n:Evidence_Chunk) REQUIRE n.chunk_id IS UNIQUE;

CREATE INDEX legal_node_node_type IF NOT EXISTS
FOR (n:LegalNode) ON (n.node_type);

CREATE INDEX legal_document_document_id IF NOT EXISTS
FOR (n:Legal_Document) ON (n.document_id);
\end{ThesisCode}

\section{Diagnostic Cypher Queries}
The following Cypher queries are executed by \texttt{scripts/inspect\_legal\_graph.py} to audit database counts:
\begin{ThesisCode}
// Count nodes by type
MATCH (n:LegalNode)

ORDER BY node_type;

// Count relationships by type

\chapter{Evaluation Metrics}
\label{app:evaluation-metrics}

This appendix provides the formal mathematical definitions of the evaluation metrics used to measure retrieval effectiveness and response compliance in Chapter 5.

\section{Information Retrieval Metrics}
Let $Q$ represent the set of benchmark queries, let $G(q)$ represent the set of ground-truth legal citations for a query $q \in Q$, and let $\mathcal{C}_k(q)$ represent the set of citations retrieved within the top-$k$ candidates.

\textbf{Recall@k}:
\[
\operatorname{Recall}@k = \frac{1}{|Q|} \sum_{q \in Q} \frac{|G(q) \cap \mathcal{C}_k(q)|}{|G(q)|}
\]

\textbf{Required Citation Coverage}:
\[
\operatorname{RequiredCitationCoverage} = \frac{1}{|Q|} \sum_{q \in Q} \mathbb{I}\left[ G(q) \subseteq \mathcal{C}_k(q) \right]
\]
where $\mathbb{I}\left[ \cdot \right]$ is the boolean indicator function.

\textbf{Forbidden Citation Violation Rate}:
Let $F(q)$ define the set of legally obsolete or distracting citations for query $q$:
\[
\operatorname{ForbiddenViolationRate} = \frac{1}{|Q|} \sum_{q \in Q} \mathbb{I}\left[ F(q) \cap \mathcal{C}_k(q) \neq \emptyset \right]
\]

\section{End-to-End System Pass Logic}
The complete end-to-end pass state for query $q \in Q$, denoted as $\operatorname{Pass}_{\mathrm{E2E}}(q)$, is computed via:
\[
\operatorname{Pass}_{\mathrm{E2E}}(q) = \operatorname{Pass}_{\mathrm{retrieval}}(q) \land \operatorname{Pass}_{\mathrm{answer}}(q) \land \operatorname{Pass}_{\mathrm{citation}}(q) \land \operatorname{Pass}_{\mathrm{quality}}(q)
\]
where each term denotes the boolean outcome of the retrieval layer, the answer presence gate, the citation containment check, and the token density quality check.
